# 伙伴增长控制台实施文档

## 1. 文档目的

这份文档的目标是把“伙伴增长控制台”从一个页面 Demo，真正变成可以给其他公司正常使用的正式系统。

这份文档会解决 3 个问题：

1. 先把业务规则写清楚，避免后面反复返工。
2. 把整个系统从 0 到上线要做的事情，一步一步拆开。
3. 用尽量通俗的方式说明每一步到底在做什么，方便非技术人员理解和推进。

---

## 2. 当前已经确认的业务规则 V1

下面这些内容，是目前已经明确的规则，后面开发都应该以这些规则为准。

### 2.1 账号和角色

- 平台不开放自由注册。
- 合作公司不能自己创建公司。
- 由我们内部先创建合作公司。
- 由我们内部创建“合作公司老板”的账户。
- 由合作公司老板创建自己公司的员工账户。

### 2.2 各角色能做什么

#### 我们内部

- 创建合作公司。
- 创建合作公司老板账户。
- 如有特殊情况，可以修改用户归因。
- 管理整个系统。

#### 合作公司老板

- 查看自己公司的全部数据。
- 添加员工。
- 管理员工账号。
- 查看团队业绩。
- 查看员工对应的用户明细。
- 导出自己公司的数据。

#### 合作公司员工

- 只看自己的邀请码。
- 只看自己的拉新数据。
- 只看自己归因下的用户明细。
- 不可以看别的员工数据。

### 2.3 用户归因规则

- 只保留 `邀请码` 归因。
- 删除 `邀请链接`。
- 不需要 `渠道码`。
- 用户通过邀请码注册后，直接绑定到对应员工。
- 一旦绑定成功，不允许换绑。
- 如果确实要修改归因，只有我们内部的人可以改。

### 2.4 分成和结算规则

- 控制台里不计算分成比例。
- 控制台里不展示分成比例。
- 控制台里不展示结算周期。
- 结算由我们和合作公司老板私下处理。
- 控制台只负责展示真实业务数据，例如：
  - 拉新人数
  - 付费人数
  - 充值总额
  - 用户明细

### 2.5 异常处理规则

- 目前主要按“充值数据”来做业绩统计。
- 因为分成只针对充值，所以这版系统先重点保证充值数据准确。
- 关于作弊、退款、洗钱等复杂问题，第一版先不放到控制台里做复杂规则判断。
- 这类特殊情况可暂时由内部人工处理。

### 2.6 数据查看范围

- 默认可查看 `全部历史数据`。
- 同时支持按时间筛选查看某一段时间内的数据。

## 3. 这个系统最终要长成什么样

正式版系统的核心目标很简单：

- 给合作公司老板一个对外控制台。
- 让老板可以管理员工和查看团队业绩。
- 给每个员工一个个人控制台。
- 让员工可以看自己的邀请码、自己的拉新、自己的用户充值明细。
- 所有数据必须隔离，老板只能看自己公司的，员工只能看自己的。

你可以把这个系统理解成：

- 表面上是一个后台页面；
- 真正核心是“账号、权限、数据归因、充值数据同步、安全控制”。

## 第 3 步：设计数据库

### 这一步要做什么

把系统需要存的数据全部设计成数据库表。

### 为什么必须做

因为系统真正运行时，所有内容都要存到数据库里：

- 老板账号
- 员工账号
- 邀请码
- 被归因用户
- 充值记录
- 操作日志

### 先说结论：第一版数据库就按 7 张主表来设计

为了让你后面能真正做出来，第一版数据库建议固定成下面 7 张表：

1. `companies`
2. `company_accounts`
3. `employees`
4. `attribution_users`
5. `recharge_orders`
6. `operation_logs`
7. `login_logs`

这 7 张表已经足够支撑第一版系统。

---

### 3.1 数据库设计原则

在正式开始写表之前，你先记住下面这几条规则。

#### 规则 1：所有业务数据都要能找到它属于哪个公司

也就是说，大部分表里都应该有：

- `company_id`

这是防止串数据最关键的字段。

#### 规则 2：所有时间字段尽量统一

建议统一用：

- `created_at`
- `updated_at`

如果这个表有特殊业务时间，再额外增加：

- `bind_time`
- `pay_time`
- `login_time`

#### 规则 3：状态字段不要乱写

像下面这些字段：

- `status`
- `role`
- `bind_status`
- `login_result`

都要提前规定死允许的值，不要随便写字符串。

#### 规则 4：第一版优先“简单可用”

第一版不要一开始就设计成特别复杂的 ERP 系统。

先做到：

- 结构清晰
- 能存数据
- 能查数据
- 能防串数据

就够了。

---

### 3.2 推荐数据库类型

如果你用 Supabase，那么底层数据库就是：

- `PostgreSQL`

所以你后面设计字段时，建议优先使用这些类型：

- `bigint`
- `text`
- `boolean`
- `numeric(18,2)`
- `timestamp with time zone`
- `jsonb`

你现在不用死记这些类型，先知道大概含义就行：

- `bigint`：长整数，适合做 ID
- `text`：字符串
- `boolean`：是否，只有 true / false
- `numeric(18,2)`：金额
- `timestamp with time zone`：时间
- `jsonb`：适合存结构化详情

---

### 3.3 表与表之间的关系

你先记住下面这张关系图的逻辑。

#### 公司和账号

- 一家合作公司，会有 1 个老板账号
- 一家合作公司，也可以有多个员工账号

所以关系是：

- `companies` 对 `company_accounts` 是一对多

#### 账号和员工

- 老板账号是账号，但不是员工
- 员工账号对应一个员工业务资料

所以关系是：

- `company_accounts` 对 `employees` 是一对一或一对零

解释一下：

- 老板账号没有员工资料
- 员工账号才有员工资料

#### 员工和归因用户

- 一个员工可以归因多个用户

所以关系是：

- `employees` 对 `attribution_users` 是一对多

#### 用户和充值订单

- 一个用户可以有多笔充值

所以关系是：

- `attribution_users.platform_user_id` 对 `recharge_orders.platform_user_id` 是一对多

### 3.4 每张表应该怎么设计

下面我直接按“表名 + 字段 + 字段含义 + 第一版规则”的方式给你定下来。

#### 1. 合作公司表 `companies`

**用途**

- 存合作公司基础信息。

**这张表解决什么问题**

- 系统里有哪些合作公司
- 每个老板属于哪个公司
- 每个员工属于哪个公司

**推荐字段**

- `id`
  - 类型：`bigint`
  - 含义：主键 ID
- `company_name`
  - 类型：`text`
  - 含义：公司名称
- `company_code`
  - 类型：`text`
  - 含义：公司唯一编码
- `status`
  - 类型：`text`
  - 含义：公司状态
  - 推荐值：`active`、`disabled`
- `remark`
  - 类型：`text`
  - 含义：备注
- `created_at`
  - 类型：`timestamp with time zone`
  - 含义：创建时间
- `updated_at`
  - 类型：`timestamp with time zone`
  - 含义：更新时间

**第一版规则**

- `company_name` 不能为空
- `company_code` 必须唯一
- `status` 默认是 `active`

**你可以这样理解**

这张表就是“合作公司总表”。

- `display_name`
  - 类型：`text`
  - 含义：页面显示姓名
- `phone`
  - 类型：`text`
  - 含义：手机号

  - 类型：`text`

#### 3. 员工信息表 `employees`

**用途**

- 存员工业务信息。

**为什么还要单独建这张表**

因为账号表是“登录身份”，员工表是“业务身份”。

比如：

- 员工账号负责登录
- 员工表负责保存邀请码和业绩归属

**推荐字段**

- `id`
  - 类型：`bigint`
  - 含义：主键 ID
- `company_id`
  - 类型：`bigint`
  - 含义：所属公司 ID
- `account_id`
  - 类型：`bigint`
  - 含义：对应的账号 ID
- `employee_name`
  - 类型：`text`
  - 含义：员工姓名
- `invite_code`
  - 类型：`text`
  - 含义：员工专属邀请码
- `status`
  - 类型：`text`
  - 含义：员工状态
  - 推荐值：`active`、`disabled`
- `created_at`
  - 类型：`timestamp with time zone`
  - 含义：创建时间
- `updated_at`
  - 类型：`timestamp with time zone`
  - 含义：更新时间

**第一版规则**

- 每个员工只保留 1 个邀请码
- `invite_code` 必须唯一
- `account_id` 必须唯一
- `company_id` 必须存在

**你可以这样理解**

这张表就是“员工业务资料表”。

---

#### 4. 被归因用户表 `attribution_users`

**用途**

- 存用户归因结果。

**这张表解决什么问题**

- 这个平台用户是谁拉来的
- 是通过哪个邀请码绑定的
- 绑定时间是什么时候
- 是否允许继续算业绩

**推荐字段**

- `id`
  - 类型：`bigint`
  - 含义：主键 ID
- `company_id`
  - 类型：`bigint`
  - 含义：所属公司 ID
- `employee_id`
  - 类型：`bigint`
  - 含义：归因员工 ID
- `platform_user_id`
  - 类型：`text`
  - 含义：你们主平台的用户 ID
- `invite_code`
  - 类型：`text`
  - 含义：绑定时使用的邀请码
- `bind_time`
  - 类型：`timestamp with time zone`
  - 含义：绑定时间
- `bind_status`
  - 类型：`text`
  - 含义：绑定状态
  - 推荐值：`bound`、`invalid`
- `created_at`
  - 类型：`timestamp with time zone`
  - 含义：创建时间
- `updated_at`
  - 类型：`timestamp with time zone`
  - 含义：更新时间

**第一版规则**

- `platform_user_id` 只能绑定 1 次
- 同一个用户不能重复归因给不同员工
- 如需调整，只能内部后台修改
- `employee_id` 和 `company_id` 必须一致

**最重要的一条**

这张表是“归因核心表”。

系统里很多统计，最后都要靠它。

## 第 4 步：做真正的登录系统

### 这一步要做什么

现在我们有数据库了，里面也有能登录的账号了（比如 `boss_nova` 和 `staff_sofia`）。

### 你需要用到什么技术？

因为我们在用 Supabase，最简单稳妥的做法是用它自带的 **Supabase Auth**（认证服务）。不过因为你目前是把账号和角色存在 `company_accounts` 业务表里，第一版为了快点跑通，我们可以先手写一个简单的 API 接口去查表验证。

### 怎么做？（小白保姆级操作）

1. **连接 Next.js 和 Supabase**
   - 我们需要在 Next.js 项目里安装 Supabase 的小工具：
     在你的 VS Code 终端里（确保在 `partnerx-app` 文件夹下），输入并回车：
     ```bash
     npm install @supabase/supabase-js
     ```

2. **配置连接钥匙**
   - 在你的项目最外层文件夹（和 `package.json` 同级）新建一个文件叫 `.env.local`。
   - 打开 Supabase 后台，点击左下角的 ⚙️ **Project Settings** -> 选择 **API**。
   - 找到 **Project URL** 和 **Project API Keys (anon public)**。
   - 把它们填进 `.env.local` 里，像这样：
     ```env
     NEXT_PUBLIC_SUPABASE_URL=你的URL复制到这里
     NEXT_PUBLIC_SUPABASE_ANON_KEY=[REDACTED]
     ```

3. **创建后端登录接口**
   我已经帮你写好了第一版的登录接口代码，你只需要复制粘贴：
   👉 **请打开同目录下的 `03_登录接口代码.md` 文件。** 照着里面的说明创建文件。

4. **改造登录页面**
   - 把你现在静态页面上的“假登录”代码，改成去调用上面那个后端接口。
   - 登录成功后，把后端返回的角色（`boss` 或 `staff`）存到浏览器的 `localStorage` 里。
   - 然后根据角色，跳到不同的页面！

- 导出数据

### 27.2 为什么日志重要

系统一旦给外部用，之后一定会有人问：

- 谁新增了这个员工？
- 谁改了归因？
- 谁导出了数据？

没有日志会很麻烦。
