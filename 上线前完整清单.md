### 1.2 修复测试数据的密码（重要！）

**状态：必须做，否则登录会失败**

**背景：** 代码已经升级为 bcrypt 加密校验密码，但测试数据库里的密码还是明文 `123456`，两者不匹配，登录会报错。

**怎么做：**

方法一（推荐）：在 Supabase SQL Editor 里执行下面这条 SQL，把所有测试账号的密码统一改成 `123456` 的 bcrypt 哈希值：

```sql
UPDATE company_accounts
SET password_hash = '$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lihC'
WHERE username IN ('boss_nova', 'staff_sofia', 'boss_blue', 'staff_alex');
```

> 上面那串 `$2a$10$...` 就是 `123456` 加密后的结果，可以直接用。

方法二：如果你以后要新建账号，密码必须先用 bcrypt 加密再存入数据库。可以用这个在线工具生成：[https://bcrypt-generator.com](https://bcrypt-generator.com)（Cost Factor 选 10）

**验证方法：** 启动项目后，用 `boss_nova` / `123456` 能成功登录。

### 1.4 配置 Row Level Security（RLS）

**状态：必须做，否则数据库有安全漏洞**

**背景：** Supabase 默认会开启 RLS，但如果你没有配置规则，前端直接访问数据库会被拒绝。我们的代码用的是服务端 `service_role key`，所以 RLS 不会影响后端接口，但为了安全，建议把 RLS 开启并且不允许前端直接读写。

**怎么做：**
1. 在 Supabase 左侧点击 `Authentication` → `Policies`
2. 对每张表，确认 RLS 是开启状态（绿色 `Enabled`）
3. 不需要添加任何 Policy（因为我们所有操作都走后端接口，不走前端直连）

**参考文档：** 本目录下的 `01_RLS提示怎么选.md`

---

### 3.2 登录失败次数限制（防暴力破解）

**状态：建议做，正式上线前完成**

**背景：** 目前登录接口没有限制，有人可以无限次尝试密码。

**最简单的做法：** 在 Supabase 里用一张表记录失败次数，连续失败 5 次就锁定 15 分钟。

或者更简单：部署到 Vercel 后，Vercel 自带基础的请求频率限制，可以先依赖这个。

---

### 3.3 登录 Token 机制（替换 localStorage）

**状态：建议做，正式上线前完成**

**背景：** 目前登录信息存在浏览器的 `localStorage` 里，这有一定安全风险（XSS 攻击可以读取）。

**更安全的做法：** 改用 `httpOnly Cookie` 存储登录态。

**如果暂时不改：** 至少确保登录信息里不存密码，只存用户 ID 和角色（目前代码已经是这样的，可以接受）。

---

### 3.4 HTTPS

**状态：部署到 Vercel 后自动解决**

Vercel 会自动给你的域名配置 HTTPS，不需要手动操作。

## 第五部分：正式使用前的准备

### 5.1 创建真实的合作公司和老板账号

**背景：** 测试数据里的公司和账号是假的，正式使用前需要创建真实的。

**怎么做：**

目前没有内部管理后台，所以需要直接在 Supabase 里手动创建。

**第一步：创建公司**
在 Supabase SQL Editor 执行：
```sql
INSERT INTO companies (company_name, status)
VALUES ('合作公司真实名称', 'active');
```
执行后，去 `Table Editor` → `companies` 表，找到刚创建的公司，记下它的 `id`（一串类似 `abc123...` 的字符）。

**第二步：生成密码哈希**
打开 [https://bcrypt-generator.com](https://bcrypt-generator.com)，输入你要设置的密码，Cost Factor 选 10，点击生成，复制结果。

**第三步：创建老板账号**
```sql
INSERT INTO company_accounts (company_id, role, username, password_hash, display_name, status)
VALUES (
  '上一步记下的公司id',
  'boss',
  '老板登录账号',
  '上一步生成的哈希值',
  '老板显示名称',
  'active'
);
```

### 5.3 和主平台打通数据（核心！）

**背景：** 控制台本身不产生数据，它只展示数据。真正的用户注册数据和充值数据来自你们的主平台。如果不打通，控制台里永远是空的。

**需要主平台做的事：**

当用户在主平台用邀请码注册时，主平台需要往控制台数据库的 `attribution_users` 表写入一条记录：
```
platform_user_id = 用户在主平台的ID
employee_id = 邀请码对应的员工ID
company_id = 员工所属公司ID
invite_code = 用户填写的邀请码
bind_time = 注册时间
bind_status = 'bound'
```

当用户在主平台充值成功时，主平台需要往 `recharge_orders` 表写入一条记录：
```
platform_user_id = 用户ID
employee_id = 该用户归因的员工ID
company_id = 员工所属公司ID
order_no = 订单号（唯一）
amount = 充值金额
status = 'success'
pay_time = 支付时间
is_first_recharge = 是否首充（true/false）
```

**打通方式（技术层面）：**
- 方式一：主平台直接用 Supabase 的 `service_role key` 写入数据（最简单）
- 方式二：主平台调用一个专门的同步接口（更安全，后续可以做）

## 附：常见问题

**Q：登录提示"账号不存在或密码错误"，但账号是对的？**
A：大概率是密码没有用 bcrypt 加密存储。执行 1.2 里的 SQL 修复。

**Q：页面打开是空白或报错？**
A：检查 `.env.local` 里的三个环境变量是否正确，重启 `npm run dev`。

**Q：Vercel 部署后登录失败？**
A：检查 Vercel 后台的 Environment Variables 是否配置了 `SUPABASE_SERVICE_ROLE_KEY`（这个变量没有 `NEXT_PUBLIC_` 前缀，容易漏掉）。

**Q：合作公司老板说看不到数据？**
A：检查主平台是否已经往 `attribution_users` 和 `recharge_orders` 表写入了数据。

**Q：如何重置某个账号的密码？**
A：在 Supabase SQL Editor 执行：
```sql
UPDATE company_accounts
SET password_hash = '新密码的bcrypt哈希值'
WHERE username = '要重置的账号名';
```
